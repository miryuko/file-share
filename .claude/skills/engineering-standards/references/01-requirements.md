# 需求分析指南

## 什么时候需要写需求文档

P0 和 P1 功能必须在编码前先写需求文档。P2 功能和 bugfix 在 GitHub Issue 里写清楚即可。

## 需求文档模板

在 `docs/features/{功能名}.md` 创建文件，四个章节缺一不可。

### 1. 场景

用"角色 → 前置条件 → 操作 → 期望结果"描述。必须覆盖正常流程和异常流程。

以剪贴板文本传输为例：

```markdown
## 场景

### 场景 1：发送文本
- 角色：发送方
- 前置条件：已打开首页
- 操作：在输入框粘贴文本内容，点击发送
- 期望：显示 6 位分享码

### 场景 2：接收文本
- 角色：接收方
- 前置条件：持有有效分享码
- 操作：在首页输入分享码，点击接收
- 期望：页面展示文本内容，提供一键复制按钮

### 异常场景
- 文本超过 1MB → 提示"文本过长，当前限制 1MB，请使用文件传输"
- 分享码不存在 → 提示"分享码无效，请检查后重试"
- 文本已过期 → 提示"内容已过期，文件保留 1 小时后自动清理"
- 服务不可用 → 提示"服务异常，请稍后重试"
```

异常场景不能少于 3 个。每条异常必须写清楚用户看到的提示文案。

### 2. 功能边界

```markdown
## 功能边界

包含：
- 粘贴文本、手动输入文本
- 生成 6 位分享码
- 接收方查看并一键复制

不包含：
- Markdown 渲染
- 代码语法高亮
- 富文本编辑
- 文本修改后重新分享

限制：
- 单条文本 ≤ 1MB
```

"不包含"和"包含"同等重要——防止范围蔓延。

### 3. 验收标准

每条必须可以直接变成测试用例。用 checklist 格式：

```markdown
## 验收标准
- [ ] 粘贴一段 URL，点击发送，生成 6 位码
- [ ] 接收方输入码，文本完整展示，无截断
- [ ] 一键复制按钮点击后内容写入系统剪贴板
- [ ] 粘贴 2MB 内容，提示"文本过长"
- [ ] 过期后扫码，提示"内容已过期"
- [ ] 输入不存在的分享码，提示"分享码无效"
```

### 4. 架构一致性检查

```markdown
## 架构对齐
- 存储位置：KV，键前缀 `text:`
- 传输路径：纯服务端中转，不涉及 P2P 和 R2
- 与现有功能的关系：复用 Cron cleanup 机制（扫描 `text:` 前缀）
- 所属 Phase：Phase 4（体验打磨）
```

作用是防止新功能跟现有架构冲突。比如如果有人提出"文本也支持 P2P"，在这里就会发现不合理。

## 完成标准

- [ ] 四个章节齐全
- [ ] 异常场景 ≥ 3 个，每条有具体用户提示文案
- [ ] 功能边界明确了"不包含"
- [ ] 验收标准可测试
- [ ] 架构对齐检查通过，不跟现有约束冲突
