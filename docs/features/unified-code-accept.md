# 统一分享码接收

## 1. 为什么做

当前接收页面要求用户**先选择类型（文件/文本），再输入分享码**。这带来三个问题：

- **认知负担**：用户拿到分享码时不一定知道它是文件还是文本，需要猜测或询问发送方
- **多余步骤**：选择类型是可在后台自动完成的操作，不应让用户手动执行
- **错误路径**：用户选错类型（选了"文件"但实际是文本分享码），会看到"分享码无效"错误，产生困惑

不做的后果：接收体验割裂，用户需要在两种模式间来回切换猜测。

## 2. 场景

### 正常场景

| 角色 | 前置条件 | 做了什么 | 期望结果 |
|------|---------|---------|---------|
| 接收者 | 拿到文件分享码 | 在接收页输入 6 位码，点接收 | 自动识别为文件，展示文件列表和下载按钮 |
| 接收者 | 拿到文本分享码 | 在接收页输入 6 位码，点接收 | 自动识别为文本，展示文本内容和复制按钮 |
| 接收者 | 从 URL 进入 `/receive/ABC123` | 页面自动查询 | 自动识别类型并展示对应 UI |

### 异常场景

| 场景 | 用户看到什么 |
|------|------------|
| 分享码不存在（两种类型都查不到） | "分享码无效，请检查后重试" |
| 分享码已过期（文件过期 + 没有对应文本） | "内容已过期" |
| 网络异常（请求失败） | "查询失败，请检查网络后重试" |
| 输入不足 6 位 | "请输入 6 位分享码" |

## 3. API 变化

### 新增：`GET /api/code/:code`

统一查询分享码，自动识别类型。

**请求**：无请求体

**响应 200（文件）**：
```json
{
  "type": "file",
  "code": "ABC123",
  "files": [{ "fileId": "file_0", "filename": "doc.pdf", "size": 1024, "contentType": "application/pdf" }],
  "expiresAt": 1715000000000,
  "remainingDownloads": 5
}
```

**响应 200（文本）**：
```json
{
  "type": "text",
  "content": "Hello World",
  "expiresAt": 1715000000000,
  "createdAt": 1714996400000
}
```

**响应 404**：`{ "code": "CODE_NOT_FOUND", "message": "分享码无效，请检查后重试" }`

**响应 410**：`{ "code": "CODE_EXPIRED", "message": "内容已过期" }`

### 保留不变

- `GET /api/session/:code` — 继续可用，向后兼容
- `GET /api/text/:code` — 继续可用，向后兼容

## 4. 数据流

```
用户输入 6 位码
       │
       ▼
GET /api/code/:code
       │
       ├─► KV 并行查询 code:<code> 和 text:<code>
       │
       ├─► code:<code> 存在且未过期 → 返回 type:"file"
       │
       ├─► text:<code> 存在且未过期 → 返回 type:"text"  
       │
       ├─► 两者都不存在 → 404
       │
       └─► 存在但已过期 → 410
```

查找优先级：文件会话 > 文本分享（如两者同时有效，返回文件类型）

## 5. 失败了会怎样

| 失败点 | 用户感知 |
|--------|---------|
| KV 读取超时 | 前端显示"查询失败，请检查网络后重试" |
| code API 返回 404 | 显示"分享码无效，请检查后重试" |
| code API 返回 410 | 显示"内容已过期" |
| 网络断开 | 前端 catch 显示通用网络错误 |

## 6. 属于哪个 Phase

Phase 4（体验打磨）— UX 优化，简化接收流程。
